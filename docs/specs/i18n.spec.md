# Spec: Internacionalização (i18n) — Chess Coach Bot

## Visão Geral

O bot opera hoje com textos hardcoded em inglês no backend (`bot/domain/messages.py`) e em português misturado com inglês no frontend da Mini App. Esta spec define a implementação de tradução automática baseada no idioma do usuário, para frontend e backend.

---

## Escopo

| Camada | Status atual | Meta |
|--------|-------------|------|
| Mini App (JS) | `messages.js` por tela, en/pt | Detectar idioma do Telegram/browser, suporte extensível a N idiomas |
| Bot (Python) | `Messages` com strings fixas em inglês | Resolver idioma por `language_code` do usuário, carregar tradução |

---

## Arquitetura

### Frontend (Mini App)

A base já está implementada em `miniapp/shared/i18n.js` e `miniapp/attack-training/messages.js`.

**Fluxo atual:**
```
Telegram.WebApp.initDataUnsafe.user.language_code
  → detectLang(Object.keys(MESSAGES))
  → T = MESSAGES[lang] ?? MESSAGES['en']
```

**Limitação:** os idiomas são embutidos no JS de cada tela. Adicionar um novo idioma exige editar cada `messages.js`.

**Solução — JSON externo por idioma:**
```
miniapp/
└── locales/
    ├── en.json     ← todos os textos do projeto em inglês
    └── pt.json     ← todos os textos do projeto em português
```

Cada arquivo JSON tem chaves planas por tela:
```json
{
  "attack.title": "⚔️ Attack Training",
  "attack.subtitle": "Tap all pieces that can be captured",
  "attack.checkBtn": "✅ Check",
  ...
}
```

`shared/i18n.js` carrega o JSON correto via `fetch` na inicialização:
```js
export async function loadLocale(lang) {
  const supported = ['en', 'pt'];
  const target = supported.includes(lang) ? lang : 'en';
  const resp = await fetch(`../locales/${target}.json`);
  return resp.json();
}
```

Cada `main.js` de tela passa a ser `async` e aguarda o locale antes de renderizar:
```js
const lang   = detectLang(['en', 'pt']);
const locale = await loadLocale(lang);
const T      = makeT(locale, 'attack');  // prefixo 'attack' para as chaves
```

`makeT` resolve chaves com prefixo e suporta templates:
```js
// locale: { "attack.correct": "🎉 Correct! {n} piece(s) identified." }
// makeT(locale, 'attack').correct({ n: 3 }) → "🎉 Correct! 3 piece(s) identified."
```

**Pluralização:**
Para idiomas como português (1 peça / 2 peças), usar a API `Intl.PluralRules`:
```js
// no locale JSON:
// "attack.correct_one": "🎉 Correto! {n} peça capturável identificada."
// "attack.correct_other": "🎉 Correto! {n} peças capturáveis identificadas."
function plural(locale, key, n) {
  const rule = new Intl.PluralRules(currentLang).select(n);
  return locale[`${key}_${rule}`] ?? locale[`${key}_other`];
}
```

### Backend (Python)

**Estado atual:** `bot/domain/messages.py` contém a classe `Messages` com atributos de classe em inglês fixo.

**Proposta:**

1. **Estrutura de arquivos:**
```
bot/domain/
└── locales/
    ├── en.json
    └── pt.json
```

2. **Classe `I18n`** em `bot/domain/i18n.py`:
```python
import json
from pathlib import Path
from functools import lru_cache

_LOCALES_DIR = Path(__file__).parent / "locales"
_SUPPORTED   = {"en", "pt"}

@lru_cache(maxsize=None)
def _load(lang: str) -> dict:
    path = _LOCALES_DIR / f"{lang}.json"
    return json.loads(path.read_text(encoding="utf-8"))

def get_messages(language_code: str) -> dict:
    lang = language_code.split("-")[0].lower() if language_code else "en"
    if lang not in _SUPPORTED:
        lang = "en"
    return _load(lang)
```

3. **Uso nos handlers:**
```python
@router.message(Command("attack-training"))
async def cmd_attack_training(message: Message, state: FSMContext):
    lang = message.from_user.language_code or "en"
    M    = get_messages(lang)

    await message.answer_photo(
        ...,
        caption=M["attack.question"],
        ...
    )
```

4. **`messages.py` como atalho de compatibilidade:**
Para não quebrar código existente durante a migração, manter `Messages` mas fazê-lo delegar para `get_messages("en")`. Remover `Messages` após todos os handlers estarem migrados.

## Critérios de aceite

### Frontend
- [ ] Usuário com `language_code: "pt"` no Telegram recebe interface em português
- [ ] Usuário com `language_code: "es"` (não suportado) recebe interface em inglês
- [ ] Adicionar um novo idioma requer apenas: criar `locales/xx.json` e adicionar `"xx"` à lista de suportados — sem tocar em `messages.js`
- [ ] Templates com variáveis e pluralização funcionam corretamente

### Backend
- [ ] `get_messages("pt")` retorna dict com chaves em português
- [ ] `get_messages("en-US")` normaliza para `"en"` e retorna inglês
- [ ] `get_messages("zh")` cai para `"en"` (idioma não suportado)
- [ ] Locale carregado com `lru_cache` — leitura de disco ocorre apenas uma vez por idioma
- [ ] Testes em `bot/tests/domain/test_i18n.py` cobrem todos os casos acima

---

## Ordem de implementação

1. **Frontend primeiro** — migrar `messages.js` para `locales/en.json` + `locales/pt.json`; atualizar `i18n.js` com `loadLocale`; atualizar `main.js` das telas para `async`
2. **Backend** — criar `bot/domain/locales/`, implementar `i18n.py`, migrar um handler como prova de conceito (`attack_training.py`)
3. **Migração gradual** — handler por handler, mantendo `Messages` como compat shim até zerar
